# 企业内部统一 AI 研发接入与治理平台设计

## 1. 文档定位

- 文档类型：产品需求与方案设计草案
- 文档日期：2026-03-18
- 适用范围：企业内部 AI 研发接入、工作区治理、权限与审计体系
- 设计目标：从零定义新的平台方案，不继承旧原型中的“项目 Key 管理中心”思路

## 2. 背景与问题

企业内研发团队正在越来越多地使用 `Claude Code`、`Codex`、`Cursor`、`CLI/CI` 等 AI 客户端开展编码、检索、排障、分析与自动化工作。现有常见问题包括：

- 企业真实上游模型凭证缺乏统一治理
- 项目成员、AI 能力、客户端接入之间缺少统一身份体系
- “Key 管理”概念过强，容易把真实供应商 Key 发放误解为平台核心能力
- 知识库、MCP、Skill、模型调用、审计与额度控制彼此割裂
- 不同客户端治理深度不同，但缺少清晰的接入边界表达

本方案的目标是建立一套面向企业内部的统一平台，使研发在客户端中感知到的是“带项目身份和权限边界的 AI 工作区能力”，而不是分散的 Key、工具和文档入口。

## 3. 产品定位

平台定位为：**企业内部统一 AI 研发接入与治理平台**。

平台不是对外售卖产品，也不是单纯的供应商 Key 发放工具。平台核心价值在于：

- 统一托管企业级真实上游模型凭证
- 统一为项目提供 AI 工作区
- 统一为研发提供可接入、可审计、可治理的 AI 能力
- 统一承接模型、知识库、MCP、Skill、额度、审批、审计等治理能力

## 4. 设计原则

### 4.1 核心原则

- 企业统一托管真实上游凭证
- 用户不直接持有组织级真实供应商 Key
- 用户获取的是项目/工作区作用域的接入凭证
- 权限默认按角色分配，个人例外必须审批并审计
- 同一工作区身份可映射到不同客户端
- 不同客户端治理深度不同，但统一纳入平台身份体系

### 4.2 架构原则

- 平台前台按“工作区接入与使用”组织
- 平台后台按“统一网关治理”建设
- “Key”不再作为一级导航核心对象
- 接入、能力、额度、审计、审批围绕工作区统一建模

## 5. 用户角色与核心任务

### 5.1 开发者

目标：在企业允许边界内，快速接入并使用 AI 能力完成研发工作。

核心任务：

- 查看自己可进入的 AI 工作区
- 查看已分配的 AI 能力
- 通过 `Claude Code`、`Codex`、`Cursor`、`CLI` 等方式接入工作区
- 使用模型、知识库、MCP、Skill 辅助研发
- 查看凭证状态、额度、使用记录、告警
- 发起例外申请，如更高额度、更多模型权限、额外工具权限

### 5.2 项目负责人 / 工作区管理员

目标：围绕项目交付，为团队分配合适 AI 能力，并控制风险与成本。

核心任务：

- 创建和维护项目下的 AI 工作区
- 配置工作区可用模型、知识库、MCP、Skill、接入方式
- 定义角色与角色对应 AI 能力集
- 将项目成员加入工作区并按角色分配能力
- 管理工作区总额度、角色额度、个人例外额度
- 处理例外申请、使用监控、超限和异常告警
- 创建和管理服务账号

### 5.3 平台管理员

目标：在企业层面统一治理 AI 资源、接入策略与审计合规。

核心任务：

- 管理企业级上游模型凭证与供应商接入
- 管理模型路由、默认策略、审计规则、默认审批策略
- 管理企业级知识资产、通用 MCP、通用 Skill、接入模板
- 查看企业范围内工作区、调用、成本、告警、风险事件
- 处理高风险事件，如凭证吊销、全局策略收紧、异常访问处置

## 6. 产品边界

平台核心边界为：**AI 研发接入治理平台**。

平台主航道覆盖：

- AI 工作区
- 模型接入治理
- 客户端接入治理
- 知识库接入
- MCP / Skill 接入
- 权限体系
- 额度体系
- 审批与审计
- 服务账号

以下能力可作为扩展模块，但不应抢主航道：

- 研发流程管理
- CI/CD 可视化平台
- 环境管理
- 事故管理
- 发布管理

## 7. 核心业务对象

### 7.1 项目与工作区

- **Project**
  - 企业内部研发项目
  - 业务与研发协作容器
- **AIWorkspace**
  - 项目下的 AI 接入与治理单元
  - 一个项目可有多个工作区
  - 系统默认在项目创建时自动创建一个默认 AI 工作区

建议支持典型工作区类型：

- 研发工作区
- 测试工作区
- 生产排障工作区

### 7.2 成员关系

- **ProjectMember**
  - 用户属于项目
- **WorkspaceMember**
  - 用户属于 AI 工作区
  - 前提必须先是项目成员

关系原则：

- 进入项目，不等于进入 AI 工作区
- 进入工作区，不等于拥有全部 AI 能力

### 7.3 角色与能力集

- **WorkspaceRole**
  - 工作区内角色
  - 支持系统预置角色与自定义角色
- **AbilitySet**
  - AI 能力集
  - 角色能力分配的核心载体

系统预置角色建议至少包含：

- 工作区所有者
- 工作区管理员
- 高级开发者
- 开发者
- 只读
- 审计者
- 安全管理员
- 服务账号角色

能力默认按角色分配，用户通常不逐人配置。

### 7.4 接入与凭证

- **ConnectionMethod**
  - 接入方式，如 `Claude Code`、`Codex`、`Cursor`、`CLI/CI`
- **ConnectionProfile**
  - 工作区在某类客户端上的接入配置模板
- **UserCredential**
  - 个人接入凭证
- **ServiceAccount**
  - 自动化、CI、MCP 等非人场景使用的服务账号
- **ServiceCredential**
  - 服务账号对应接入凭证

设计上应强调“接入凭证/连接配置”，而非“向用户发真实 Key”。

### 7.5 治理对象

- **QuotaPolicy**
  - 工作区、角色、个人三级额度策略
- **ApprovalRequest**
  - 例外权限、额度、长期凭证等审批请求
- **AuditEvent**
  - 调用、配置、权限、凭证、策略等审计事件
- **UsageRecord**
  - token、成本、模型、用户、工作区、客户端使用记录
- **AlertEvent**
  - 超限、异常调用、权限风险、凭证风险等告警事件

## 8. 权限模型

### 8.1 权限分配原则

- 默认按角色分配
- 允许个人例外覆盖
- 个人例外必须审批
- 个人例外必须进入审计

### 8.2 权限域

工作区权限域采用标准型设计，包含以下 8 类：

- 模型权限
- 知识库权限
- MCP / Skill 权限
- 客户端接入权限
- 额度权限
- 服务账号权限
- 审计权限
- 审批权限

### 8.3 个人例外原则

允许存在少量个人例外，如：

- 更高额度
- 更高模型权限
- 额外工具权限
- 长期接入凭证

例外必须具备：

- 审批来源
- 生效范围
- 有效期
- 变更记录
- 审计记录

## 9. AI 能力模型

AI 能力不向开发者暴露为“让用户自己拼上下文包”的形式，而是由工作区管理员预先定义并按角色分配。

对开发者展示为：

- 已分配 AI 能力
- 已开放模型
- 已开放知识库
- 已开放 MCP / Skill
- 已开放接入方式

内部建模上，可保留“能力集/上下文包”概念，但产品表达应优先使用：

- AI 能力集
- 已分配能力
- 接入范围

## 10. 额度与成本体系

### 10.1 额度层级

平台采用三级额度模型：

- 工作区总额度
- 角色默认额度
- 个人例外额度

### 10.2 度量维度

额度与消耗应同时覆盖：

- token
- 金额成本
- 模型级消耗明细

### 10.3 超额控制策略

达到阈值时，平台按顺序执行：

- 预警提醒
- 软限制
- 硬阻断

限制策略可作用于：

- 高成本模型
- 个人例外权限
- 工作区整体资源使用
- 高风险能力使用

## 11. 审批与审计策略建议

审批方案在本阶段先按建议值暂定。

### 11.1 默认无需审批

- 角色内默认能力使用
- 角色内默认额度使用
- 正常客户端接入
- 常规知识库与工具使用

### 11.2 默认需要审批

- 个人例外权限
- 个人例外额度
- 服务账号创建
- 长期接入凭证
- 高成本模型权限
- 批量权限变更
- 工作区关键策略变更

### 11.3 审批等级建议

- 普通风险：一级审批
- 高风险：二级审批

所有审批行为均应进入审计。

### 11.4 审计范围

平台应记录：

- 谁在什么时间通过什么客户端进入哪个工作区
- 使用了哪些模型、知识库、工具
- 消耗了多少 token 和成本
- 做了哪些权限、额度、服务账号、策略变更
- 是否触发了风险规则或超限规则

## 12. 客户端接入与治理边界

### 12.1 统一原则

- 所有客户端接入统一工作区身份体系
- 平台按客户端能力不同，采用不同治理深度
- 平台统一负责身份绑定、接入材料生成、能力边界说明、审计范围说明
- 不承诺所有客户端都达到相同深度的全链路治理

### 12.2 接入模式

建议定义三类：

- **全托管**
  - 模型请求经过平台
  - 工具请求经过平台或受平台控制
  - 权限、额度、审计、策略完整生效

- **半托管**
  - 平台统一承接工作区身份、知识库、MCP/Skill、部分模型接入
  - 客户端原生能力不一定完全可控
  - 审计与额度只覆盖平台可见范围

- **外部接入**
  - 平台主要提供接入说明、身份绑定、有限配置和治理提示
  - 不承诺全链路控制

### 12.3 客户端建议边界

#### Claude Code

- 接入模式：全托管
- 平台提供：
  - 接入配置
  - 接入凭证
  - MCP 配置
  - 工作区能力说明
  - 完整审计与用量可见性

#### Codex

- 接入模式：全托管或高托管
- 平台提供：
  - 工作区接入配置
  - 项目作用域身份
  - MCP / 工具接入说明
  - 审计与额度控制

#### Cursor

- 接入模式：半托管
- 平台提供：
  - 工作区身份
  - 知识库与 MCP / Skill
  - 接入配置与能力说明
  - 平台可见范围内的审计

需求文档中应明确：

- 不保证客户端所有原生 AI 能力都受平台全量治理
- 平台审计与控制范围需显式标注

#### CLI / CI / 自动化任务

- 接入模式：全托管
- 主要通过服务账号和项目作用域凭证接入
- 平台需完整支持：
  - 服务账号
  - 凭证生命周期
  - 审计
  - 配额
  - 风险限制

### 12.4 平台展示原则

平台不强调推荐某个客户端品牌，而是统一展示：

- 当前工作区支持哪些接入方式
- 每种接入方式可获得什么能力
- 接入模式是什么
- 凭证状态如何
- 审计覆盖范围如何

## 13. 核心业务流程

### 13.1 项目与工作区建立

- 创建项目
- 自动创建默认 AI 工作区
- 项目负责人按需新增更多工作区

### 13.2 工作区能力配置

工作区管理员配置：

- 模型范围
- 知识库范围
- MCP / Skill 范围
- 支持的客户端接入方式
- 工作区总额度
- 角色默认额度

### 13.3 角色与能力分配

- 配置角色
- 配置角色对应 AI 能力集
- 将项目成员加入工作区
- 按角色分配能力
- 少量个人例外通过审批覆盖

### 13.4 开发者接入

开发者查看：

- 自己已进入的工作区
- 自己已分配的能力
- 当前可用接入方式
- 凭证状态
- 当前额度与使用情况

开发者选择客户端后，平台生成该客户端对应接入材料。

### 13.5 开发者实际使用

对于全托管客户端，平台负责：

- 识别工作区身份
- 校验权限
- 装配工作区允许上下文
- 挂接可用 MCP / Skill
- 记账、审计、告警
- 转发到上游模型

对于半托管客户端，平台至少统一承接：

- 工作区身份
- MCP / Skill
- 知识库入口
- 部分审计

### 13.6 额度与策略控制

平台持续按工作区、角色、个人维度监控：

- token 消耗
- 成本金额
- 模型级消耗明细

达到阈值后执行：

- 提醒
- 软限制
- 硬阻断

### 13.7 例外申请与审批

当开发者需要超出角色默认边界时，可发起申请：

- 更高额度
- 更高模型权限
- 额外工具权限
- 长期接入凭证

审批通过后形成个人例外配置，并进入审计。

## 14. 信息架构建议

平台按三层组织：

### 14.1 企业层

面向平台管理员，建议模块：

- 企业 AI 资源
- 企业治理
- 企业能力资产
- 项目与工作区总览

### 14.2 项目层

面向项目负责人，建议模块：

- 项目概览
- AI 工作区
- 项目成员
- 项目知识资产
- 项目治理

### 14.3 工作区层

面向开发者与工作区管理员，建议模块：

- 工作区概览
- 接入方式
- 我的 AI 能力
- 额度与使用
- 成员与角色
- 知识与工具
- 服务账号
- 审计与审批
- 工作区设置

## 15. 导航与页面表达原则

- 不再设置“项目空间 Key 管理中心”作为核心页面
- “Key”不作为前台一级导航对象
- 开发者看到的是：
  - 工作区
  - 接入方式
  - 我的能力
  - 我的额度
- 管理员看到的是：
  - 角色
  - 能力集
  - 服务账号
  - 策略
  - 审计

真正的凭证管理应收敛到：

- 企业层的真实凭证治理
- 工作区层的接入凭证与服务账号管理

## 16. 本方案的关键结论

- 平台不再围绕“给项目发 Key”建模
- 平台围绕“项目下的 AI 工作区”建模
- 开发者使用的是被授权的工作区 AI 能力，而不是直接持有真实供应商 Key
- 工作区是权限、接入、能力、额度、审计的统一容器
- 平台应统一身份体系，但接受不同客户端治理深度不同的现实
- `Claude Code`、`Codex`、`Cursor`、`CLI/CI` 均应纳入统一接入体系

## 17. 后续建议

建议后续工作按以下顺序推进：

1. 基于本设计补充页面级需求清单
2. 输出信息架构图与页面清单
3. 输出高保真原型或中保真原型
4. 继续拆分后台对象模型与接口模型
5. 再进入实施计划编写
